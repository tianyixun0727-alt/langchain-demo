### 2.2_messages.py — 消息体系

**这个文件做什么：** 完整演示 LangChain 的四种消息类型及其用法。

**四种消息角色：**

| 角色 | 用途 | 写法 |
|------|------|------|
| SystemMessage | 控制行为风格 | ("system", "...") 或 SystemMessage(...) |
| HumanMessage | 用户输入 | ("human", "...") 或 HumanMessage(...) |
| AIMessage | 模型回复 | AIMessage(...) |
| ToolMessage | 工具返回结果 | **必须用对象**，需指定 	ool_call_id |

**重点讲工具调用流程：**

这是最容易出错的地方。当模型决定调用工具时，消息流是这样的：

`
HumanMessage（用户提问）
  → AIMessage（模型说"我要调用工具"，带 tool_calls）
    → ToolMessage（工具返回结果，tool_call_id 必须匹配）
      → 模型自动生成最终回复（不要再手动插 AIMessage！）
`

**常见踩坑：**
- ToolMessage 必须用对象写法，元组写法无法指定 	ool_call_id
- 	ool_call_id 必须和 AIMessage 里的 id 精确匹配

**这个文件做什么：** 对比模型级流式和 Agent 级流式。

### 3.1_prebuilt.py — 预置中间件：自动摘要

**这个文件做什么：** 解决"对话太长导致 token 超限"的问题。

**问题场景：** 用户和 Agent 聊了 50 轮，消息越来越长，每次调模型的 token 开销越来越大，最终超出上下文窗口。

**工作原理：** 当对话超过阈值时，自动把历史消息压缩成摘要，只保留最近的消息。模型看到的是"摘要 + 最近对话"，既不丢上下文，又不会爆 token。


最后一条提问“我之前问过什么？”，AI 完整列出了之前提到的 全部 5 个要点，一条没漏。

说明中间件确实把长对话压缩成了有效摘要，并成功保留下来。

### 3.2.1_Node-style hooks.py — 节点式钩子

**这个文件做什么：** 在模型调用前后插入自定义逻辑（如日志记录）。

**两种钩子装饰器：**

``python
@before_model
统计当前对话轮数

@after_model
回复长度和回复预览

**特点：** 在流程的某个固定位置"插一段代码"，只观察、不干预。适合日志记录、监控、审计。

---

### 3.2.2_Wrap-style hooks.py — 包装式钩子

**这个文件做什么：** 接管整个模型调用过程，实现指数退避重试。


**和节点式钩子的区别：**

#节点式钩子和包装式钩子的最大区别在于控制范围不同。
节点式钩子是在某个固定节点前后执行额外逻辑，比如 before_model 和 after_model，通常用于日志记录、监控和状态修改；
而包装式钩子会把整个模型调用过程包裹起来，能够拦截并控制整个调用链，因此可以实现自动重试、限流、熔断、模型切换等更高级的功能。
简单来说，节点式钩子是“在流程某个位置插一段代码”，而包装式钩子是“接管整个流程并决定它怎么执行”。

### 4.1_guardrails — 安全机制：PII 脱敏

**这个文件做什么：** 自动检测并遮盖用户输入中的个人信息（PII）。

``python
agent = create_agent(
    model=llm, tools=[],
    middleware=[PIIMiddleware("email", strategy="redact", apply_to_input=True)]
)
``

**场景：** 用户说 "My email is john@example.com"，PIIMiddleware 会在发给模型之前把邮箱替换成 [REDACTED]，防止敏感数据泄露给模型服务。

**为什么重要：** 在企业场景中，用户输入可能包含身份证号、手机号、邮箱等敏感信息。不脱敏就发到模型 API，是合规风险。

### 4.2-runtime — 运行时系统：依赖注入

### 4.3-context-engineering — 上下文工程（三个 Demo）

上下文工程是 Agent 开发中**最容易被忽视、但对效果影响最大**的环节。它解决的核心问题是：**给模型看什么信息、以什么形式看、在什么时候看。**

---

#### 4.3.1 Model context — 动态系统提示词

**问题：** 固定的 system prompt 不够灵活。短对话需要友好，长对话需要简洁。

**解决方案：**


每次模型调用前，动态生成 system prompt。对话短的时候详细回答，对话长的时候压缩回答，避免 token 浪费。

---

#### 4.3.2 Tool context — 工具读写状态

**问题：** 工具不仅是"执行任务"，还需要能读写 Agent 的状态。

比如一个购物车工具：添加商品时，它不能只返回“添加成功”，还得知道当前购物车里已经有什么，才能算出总价。
这就意味着——工具需要读写状态。

配合 InMemorySaver 和 	hread_id，状态还能跨调用持久化。第一次调用记录 visit_count=1，第二次变成 2，第三次变成 3。

---

#### 4.3.3 Life-cycle context — 生命周期自动摘要

**这个文件和 3.1 的 SummarizationMiddleware 类似**
在 3.1 中我们关注的是“预置中间件怎么用”；
而在这里，我们要看到的是 “生命周期”——它发生在 Agent 执行流程的哪个阶段。

中间件会在 Agent 执行的特定节点（如消息到达后、模型调用前）自动触发，无需工具主动调用。

SummarizationMiddleware 就是在 消息历史累积到一定 token 时自动介入，把早期内容压缩成摘要，整个过程对开发者透明。

### 4.5-hitl — 人机协作（Human-in-the-Loop）

**这个文件做什么：** 邮件发送审批流程——Agent 想发邮件时，必须人工确认。

#### 4.6.6 Custom Workflow — 自定义工作流

**核心思路：** 用 LangGraph 的 StateGraph 构建完全自定义的流程图。

``python
class GraphState(TypedDict):
    input: str
    step: str
    output: str

graph = StateGraph(GraphState)
graph.add_node("classify", classify)
graph.add_node("weather", weather_node)
graph.add_conditional_edges("classify", router, {...})
``

**特点：** 完全确定性——分类结果决定走哪条边，不存在 LLM 决策的不确定性。适合对流程有严格要求的场景。

**五种模式的演进：** 从简单到复杂——Subagents（1 对多调度）→ Handoffs（互相转交）→ Skills（单 Agent 多技能）→ Router（分类分发）→ Custom Workflow（完全自定义）。

### 4.7-rag — 检索增强生成（三个 Demo）

RAG 是目前让 LLM 利用外部知识最成熟的技术方案。这里展示了三种递进的实现方式。

---

#### 4.7.1 2-Step RAG — 基础流程
Qdrant 是一个高性能的开源向量数据库，专门用于存储和检索高维嵌入向量。在我们的 RAG 流程中，它通过余弦相似度快速匹配知识片段，并原生支持分数阈值过滤，确保只返回高相关的文档。

**固定两步走：检索 → 生成。**

第一步：初始化模型。

我们配置了 ChatOpenAI 作为生成模型，OpenAIEmbeddings 作为向量化模型。

第二步：读取知识库。

从 knowledge_base.txt 逐行读取，每行按 | 分割，得到文档 ID 和内容。

第三步：清空旧集合。

检查 Qdrant 中是否已存在同名 collection，若有则删除，确保环境干净。

第四步：创建向量库。

使用 Qdrant.from_texts 将文本和元数据写入 Qdrant（本地 localhost:6333），使用 Cosine 距离。

第五步：定义检索函数。

调用 similarity_search_with_score 并传入 score_threshold 参数，原生过滤低分文档，只返回相似度高于阈值的结果。

第六步：用户提问。

模拟一条用户问题“苹果是一种常见水果...”。

第七步：检索。

用 retrieve 函数取回 top-K 文档，打印相似度分数。

第八步：生成答案。

将检索到的文档拼接成上下文，构造 system 消息，调用 LLM 生成最终回答。


**特点：** 流程固定、简单直接。缺点是"不管需不需要，都先检索"。

---

#### 4.7.2 Agentic RAG — Agent 自主决策

**核心改变：** 把检索封装成工具，让 Agent 自己决定要不要检索。

``python
@tool
def search_knowledge_base(query: str) -> str: ...

agent = create_agent(model=llm, tools=[search_knowledge_base])
``

**效果：**
- 知识库问题 → Agent 调用检索工具
- 常识问题（"1+1=?"）→ Agent 直接回答，不浪费检索

**优点：** 更智能，减少不必要的检索开销。

---

#### 4.7.3 Hybrid RAG — 查询改写 + 重排序

**在 Agentic RAG 基础上再加两层增强：**

**第一层 — 查询改写：** 用户问"如何减少大模型幻觉？"，LLM 生成多个变体问题，提高召回率。

**第二层 — 重排序：** 检索出多个候选文档后，用 LLM 按相关性重新排序，取最相关的。

`
原始问题 → 查询改写 → 多查询检索 → 合并去重 → LLM 重排序 → 取 Top-K → 生成

### 4.8-long-term-memory — 长期记忆

**这个文件做什么：** 实现跨会话的记忆持久化。

**和短期记忆的区别：**

| 类型 | 存储 | 范围 | 实现 |
|------|------|------|------|
| 短期记忆 | InMemorySaver | 同一 thread_id 内 | 保存对话历史 |
| 长期记忆 | InMemoryStore | 跨 thread_id，按用户隔离 | 保存结构化事实 |

**运行效果：** 会话 1 中记住"我喜欢 Python"，会话 2（不同 thread_id）中问"你还记得关于我的什么？"，Agent 能回忆起来。

**关键：** 用 
amespace 按用户隔离记忆，不同用户的数据互不干扰。

### 5.1_testing.py — 五种测试策略

**为什么需要测试 Agent？** Agent 的行为不是确定性的——同样的输入可能得到不同的输出。所以测试策略和传统软件不一样。

**五种方法：**

| 类型 | 测什么 | 怎么测 |
|------|--------|--------|
| 正确性 | 答案是否准确 | 期望关键词匹配 |
| 延迟 | 响应够不够快 | 统计平均延迟和 P95 |
| 工具调用 | 调对工具了吗 | 拦截 tool_calls 比对 |
| 安全 | 会不会被注入 | 注入攻击测试 |
| 回归 | 新版本有没有退化 | 历史评分对比 |

**重点提醒：** Agent 测试不是"跑一次看结果"，而是要建立**持续回归的基线**。每次改 prompt、换模型、加工具，都要跑一遍测试套件，确保没有退化。

### 6.
与常规 RAG 不同，这里不是简单地“检索 + 生成”，而是让 Agent 自主决定何时检索、如何整合信息，并严格按照法规条文来回答
下面我们按代码顺序，一步步深入了解一下
。
第一步：配置 LLM 和嵌入模型
我们使用 deepseek-v3 作为生成模型，bge-m3 作为嵌入模型，用于后续向量化。temperature=0 保证回答的确定性，适合法规类严肃场景。

第二步：加载并分割文档
用 Docx2txtLoader 读取 Word 文档。
按 ### =<章节> 标记识别章节，将内容拆成章节块。
再用 RecursiveCharacterTextSplitter 按 500 字符切分，生成可检索的文档块，并记录章节元数据。
若识别不到章节，则回退到整篇分割。

第三步：构建 Qdrant 向量库
先检查并删除同名 collection，保证环境干净。
调用 Qdrant.from_documents 将文档块向量化并存入 Qdrant（本地 localhost:6333）。
使用余弦相似度作为距离度量，适合文本检索。
最后获取 collection 中的向量数量，确认入库成功。


第四步：定义检索工具
similarity_search_with_score 取回 top‑3 文档块，并拼接成包含章节信息的字符串，供 Agent 调用。返回一个字典列表，包含 content、chapter、score。
定义 search_regulations这个工具被 Agent 调用，作用是对用户问题执行检索，并返回结构化的条文片段（包含章节和内容）。
它将被注入到 Agent 的工具列表中，Agent 在需要时会主动调用它。


第五步：创建 Agent
使用 create_agent，挂载检索工具，并注入详细的系统提示词。
系统提示词规定了角色、技能、限制，强调 必须通过工具检索，严格依据条文回答，且回答后需注明具体章节。
这种设计确保了 回答的可追溯性 和 合规性，避免大模型“幻觉”。

第六步：交互式问答
主循环接收用户输入，调用 agent.invoke，打印助手最终回答。
用户提问时，Agent 自主决定是否调用 search_regulations，若问题超出法规范围则礼貌拒绝。



## 总结

我们今天从最简单的环境检测，一路走到了部署上线。回顾一下核心要点：

**基础三件套：** Model + Tool + Prompt → Agent

**中间件体系：**
- 预置中间件（自动摘要、PII 脱敏）
- 节点式钩子（日志、监控）
- 包装式钩子（重试、限流）

**上下文工程：** 模型上下文 → 工具上下文 → 生命周期上下文，确保模型始终看到最合适的信息

**MCP 协议：** 标准化工具服务，支持无状态、有状态·2

**多智能体：** Subagents → Handoffs → Skills → Router → Custom Workflow，按需选择

**RAG：** 2-Step → Agentic → Hybrid，逐步增强检索效果

**记忆：** 短期（InMemorySaver + thread_id）→ 长期（InMemoryStore + 用户命名空间）

**测试与部署：** 五种测试策略 + 自动评估 + HTTP 服务部署

最后给大家一个建议：**不要一上来就用最复杂的方案。** 先从 create_agent 开始，用最简单的工具和提示词把原型跑通，然后根据实际需要逐步加中间件、加记忆、加多智能体。LangChain 的设计就是"渐进式复杂度"——你不需要一步到位，但你需要知道每一步能走向哪里。
